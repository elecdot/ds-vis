    Docs Workflow — 文档仓库交付规范

    目标：把 docs/ 变成“可交付、可展示、可协作”的文档仓库（mkdocs + mermaid），并把协议/状态变化可追踪地沉淀下来。

    1. 验收标准（可执行）

    1.1 可构建（Buildable）

    • 本地可执行 mkdocs build（或 uv run mkdocs build）并无报错。
    • 站点导航结构清晰，重要页面能从 nav 直接访问（Quickstart / Project State / Design / Engineering / Diagrams / Demo）。

    1.2 不漂移（Non-drifting）

    • docs/project_state.md 作为 SSOT：其它文档不重复“当前状态”，只绑定 phase 或描述规范。
    • docs/index.md（Registry）与各文档 front matter 一致（bound_phase/version/status/last_updated）。
    • docs/engineering/dev_kb.md 记录已知问题时不与 SSOT 冲突；若冲突，以 SSOT 为准并修订 dev_kb。

    1.3 可展示（Presentable）

    • 至少提供 1 个“演示入口页”（Demo/Scenarios）：包含可直接复制的 DSL/JSON 示例与配套架构/流程图（mermaid）。
    • 关键概念有图示：分层架构、Command→Timeline 流水线、Ops/Step/Timeline 关系。

    1.4 可协作（Collaborative）

    • 新增/修改协议（例如新 OpCode、新 Command schema、新 kind 路由）时：同步更新对应设计文档 + Registry（版本号与日期）。
    • 文档中尽量提供“复现命令/脚本”而不是只给结论（如 diagrams 生成方式）。

    2. 文档写作约定（中英混排）

    • 标题：英文为主（便于导航与检索），正文可中英混排（优先清晰）。
    • 术语：首次出现给出英文括注，例如“时间线（Timeline）”“微步骤（micro-step）”。
    • 链接：站内链接优先使用相对路径（mkdocs 能正确解析）。

    3. Mermaid 约定（mkdocs）

    使用围栏代码块：

    ```mermaid
flowchart TD
  A --> B
```

    建议优先输出： - flowchart：流程/模块关系 - sequenceDiagram：一次命令执行流水线 - stateDiagram-v2：状态机（如播放控制、节点状态）

    4. 文档版本与绑定（Registry + front matter）

    当你改动下列内容时，必须同步文档： - src/ds_vis/core/ops/**：更新 docs/design/ops_spec.md，并在 docs/index.md 提升 version/last_updated。 - src/ds_vis/core/scene/**：更新 docs/design/scene_graph.md。 - 任何跨层语义变化：优先更新 docs/design/architecture.md 与 docs/design/animation.md。

    仅修正文案/拼写（不涉及语义变更）： - 可以只提升 patch version（如 v0.4.2 -> v0.4.3）或仅更新 last_updated（按团队约定）。
